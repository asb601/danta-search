1. System Overview — Layers

The system is built in four independent layers. Each layer has one job. The LLM only touches Layer 3 (SQL writing). All correctness guarantees come from Layers 1, 2, and 4.

🧑 USER / CLIENT  —  sends natural language question via chat UI
↓
🔐 AUTH + PERMISSION RESOLVER  —  JWT decode → role check → build access set
super_admin: access_set = ALL  |  domain_user: access_set = owned + granted containers
↓
🔍 MULTI-STRATEGY RETRIEVAL ENGINE  —  finds the RIGHT files from 1M+
Temporal parse → Domain filter → Date filter → BM25 → Fuzzy → Vector → Graph → Rank fusion → Top 20
↓
🤖 LANGGRAPH AGENT + GPT-4  —  reads 20 file summaries → writes SQL ONLY
LLM never answers from memory. Every number MUST come from a tool call.
↓
⚙️ DUCKDB EXECUTION  —  runs SQL on Parquet files in Azure Blob Storage
Returns exact numbers + row count + excluded rows (nulls/junk)
↓
🗄️ STORAGE LAYER
Postgres (Neon): metadata, auth, embeddings, relationships  |  Azure Blob: Parquet files  |  DuckDB: runtime query engine
The core principle: We provide the LLM with (a) the correct files and (b) their real schemas. The LLM provides (a + b) by writing SQL. DuckDB computes the answer. We never trust the LLM to recall data values from its training weights.

2. Ingestion Pipeline — The Foundation of Everything

Ingestion is the most important step in the entire system. If ingestion does its job well, every other part — retrieval, SQL generation, accuracy — works correctly. If ingestion is weak, no amount of clever AI can fix bad data downstream.

When a file arrives (CSV, Excel, or a text file full of comma-separated values), it goes through 9 sequential stages before it is ever queryable. Each stage exists to solve a specific real-world problem with messy enterprise data.

Why do we need all these stages? Because real enterprise files are never clean. A file named report.csv might actually be Windows-1252 encoded, use semicolons as delimiters, have three Excel sheets, contain "N/A" instead of real nulls, come from a 2 GB export, and have new columns added compared to last month. Every stage below handles one of these realities.
0
Encoding Detection
What it does: Before opening the file, we detect its character encoding using the chardet library — reads the first 8 KB and identifies whether the file is UTF-8, Latin-1, Windows-1252, or another format.

Why it matters: A file encoded in Windows-1252 opened as UTF-8 produces corrupted characters (called mojibake). Example: "café" becomes "cafÃ©". Without this step, the column values are unreadable and AI descriptions are wrong. This runs before anything else — no parsing until encoding is confirmed.
↓
1
File Type and Delimiter Detection
What it does: Checks the file extension and reads the first few bytes (magic bytes) to confirm the actual file type. For .txt files, Python's built-in csv.Sniffer reads the first 8 KB and detects what character separates the columns — comma, semicolon, tab, or pipe.

Why it matters: The client said some files are .txt files that are actually comma-separated data. A file named export.txt might be perfectly valid tabular data. Without sniffing, we would reject it or read it as one long string. After this step, a .txt file with detected delimiters is treated exactly the same as a .csv file.
↓
2
Excel Multi-Sheet Splitting
What it does: For .xlsx and .xls files, pandas opens the workbook and reads every sheet. Each sheet becomes a separate, independent File record in our system with its own schema, metadata, and Parquet file. The sheet name and the original filename are both stored so we know they belong together.

Why it matters: An Excel file with 5 sheets contains 5 different datasets. If we treated the whole file as one unit, we could not query "Sheet 3 — Budget Forecast" independently. Splitting enables the AI to discover that Sheet 2 of employees.xlsx contains the column it needs, even if the other sheets do not.
↓
3
Schema Inference and Version Diff
What it does: DuckDB reads a sample of the file and infers the column names and data types (integer, float, string, date). If this file was uploaded before, we compare the new column list against the previous version — this is a schema diff.

Why it matters: Data files grow. payroll_2025.csv might gain a bonus column next quarter. We do not re-process the entire file — we detect only the new columns and run additional work just for those. Old queries still work because the previous Parquet version is kept. Schema changes are handled automatically with no manual migration required.
↓
4
Dirty Data Profiling
What it does: For every column, we scan the full data and compute: percentage of null values, percentage of junk values (values that do not match the inferred type), a list of null-alias strings found in the data (words like "N/A", "TBD", "not applicable", "-" that mean null but are stored as text), and up to 3 sample junk values. This profile is stored as a JSON object in the database.

Why it matters: The client explicitly said files will contain junk and described values. Without this step, a SQL query like SUM(salary) would silently fail or give wrong answers when half the salary column contains "pending" as a string. With this profile, the AI knows in advance that salary has 8% nulls and generates WHERE salary IS NOT NULL AND salary ~ '^[0-9.]+' automatically.
↓
5
Parquet Conversion and Storage Tiering
What it does: The cleaned data is converted from CSV/Excel to Parquet format and uploaded to Azure Blob Storage. Parquet is a columnar binary format — reading a single column from a 1 GB Parquet file takes milliseconds because it skips the other columns entirely. Based on file size, we assign a storage tier: Hot (< 10 MB, cached locally), Warm (10 MB–500 MB, reads from Azure), Cold ( > 500 MB, summary Parquet generated for fast queries), Archive (historical files rarely accessed).

Why it matters: CSV files cannot be queried efficiently at scale. A 2 GB CSV read sequentially to find one column takes minutes. The same data as Parquet, queried column-by-column, takes under 2 seconds. This conversion is what makes DuckDB-powered SQL fast on large files.
↓
6
AI Description and Date Range Extraction
What it does: GPT-4 is called once per file with the column names, data types, a sample of rows, and the dirty-data profile. It writes a plain-English description of what this file contains. Separately, we scan every column that looks like a date and extract the actual minimum and maximum date in the data — stored as date_range_start and date_range_end.

Why it matters: The description is what the retrieval system searches against when a user asks a question. Good descriptions mean better file matching. The date range is what makes temporal queries work — "last week" filters files by their actual stored date range, not by guessing from the file name.
↓
7
Relationship Detection
What it does: We compute a hash from the file's top column names and find other files in the same domain that share a similar column signature. Files that share one or more column names (like emp_id or department) get a relationship edge inserted into the file_relationships table with a confidence score.

Why it matters: This is how the system knows that payroll.csv and employees.csv can be joined — they both have an emp_id column. When the AI retrieves payroll.csv for a query, the graph layer automatically adds employees.csv as a related file. The AI can then write a JOIN query without the user having to know both files exist. Naive comparison (every file vs every file) would be 500 billion comparisons at 1M files. Column-hash clustering reduces this to a few thousand comparisons per file.
↓
8
Embedding Generation
What it does: The AI-generated description plus the column names are sent to OpenAI's text-embedding-3-small model, which returns a vector of 1,536 floating-point numbers that represent the semantic meaning of the file. This vector is stored directly in the files table using the pgvector Postgres extension.

Why it matters: This is what makes semantic search possible. When a user asks "how much did we pay staff?", we convert that question to a vector and find the files whose vectors are closest in meaning. Without this step, we would have to load all 1 million file descriptions into the prompt — which is physically impossible.

What Happens When a File Is Re-Uploaded With New Columns

Schema changes are handled automatically. Below is what happens specifically when payroll_2025.csv gains two new columns on re-upload:

payroll_2025.csv — Version 1 (already ingested)

emp_id  integer
department  string
salary  float
join_date  date

payroll_2025.csv — Version 2 (new upload)

emp_id  integer
department  string
salary  float
join_date  date
bonus  float  ← NEW COLUMN
tax_bracket  string  ← NEW COLUMN

SYSTEM ACTIONS — Only for the new columns

Columns emp_id, department, salary, join_date — skipped, no reprocessing needed
Add bonus and tax_bracket to the stored schema in FileMetadata
Run relationship detection for the new column names only
Regenerate the AI description (it now includes bonus information)
Regenerate the embedding (the new description changes the meaning vector)
Archive the Version 1 Parquet — old queries that do not reference bonus still work
Write Version 2 Parquet with all 6 columns — new queries can use bonus
Ingestion is fully automatic. The admin uploads the file (or connects an Azure container). The system handles everything else — encoding, dirty data, schema versioning, relationship discovery, and search indexing — without any manual steps.

3. Why PostgreSQL — and Why It Does Not Touch the Client's Database

First: How Does the Client's Data Actually Get Into danta-search?

This is an important point to understand clearly. danta-search never connects to the client's production database. The client does not give us a MongoDB connection string, a SQL Server password, or any direct access to their live systems.

What the client provides:
One thing only — an Azure Blob Storage container connection string (or an Azure Storage Account name + SAS token).

Their files live in Azure Blob Storage. danta-search connects to that container, reads the files from the blob, and processes them. The client's MongoDB, their internal SQL Server, their application database — none of that is ever touched.

The client's existing database has zero risk. We are read-only on the blob storage. We do not write anything back to the client's systems. Their production MongoDB continues to run exactly as it does today. danta-search is an independent intelligence layer on top of exported data.

Why danta-search Uses PostgreSQL for Its Own Storage

danta-search uses Postgres to store its own internal data — file metadata, user accounts, permissions, conversation history, relationships, and embeddings. This is completely separate from the client's data. Here is why Postgres is the right choice for this internal store:

PostgreSQL — What We Use
  • pgvector extension — stores and searches 1,536-dimensional embedding vectors natively. MongoDB has no equivalent at this quality.
  • JSONB columns — stores dirty-data profiles, column schemas, and analytics as flexible JSON. Same flexibility as MongoDB documents, but inside a relational database.
  • Strong transactions — when ingestion writes to 4 tables at once (files, metadata, analytics, relationships), either all succeed or all roll back. No partial writes.
  • Recursive CTEs — can traverse file relationship graphs without a separate graph database.
  • Full-text search (tsvector) — BM25 keyword search built in. No separate search index needed.
  • pg_trgm extension — trigram-based fuzzy matching for typo-tolerant column name search. Built in.
  • SQLAlchemy ORM — our codebase is entirely built on SQLAlchemy. Switching ORMs is months of work for no benefit.
MongoDB — Why It Does Not Fit
  • No pgvector equivalent — MongoDB Atlas Search has basic vector search but does not support HNSW indexing at the quality needed for 1M file embeddings.
  • Weak transactions — multi-document transactions in MongoDB are slower and more complex. Our 9-stage ingestion pipeline writes to multiple collections atomically — this is painful in MongoDB.
  • No recursive graph traversal — MongoDB's $graphLookup is limited to one collection and is significantly slower than Postgres recursive CTEs for our relationship graph.
  • No built-in BM25 or fuzzy search — requires Atlas Search (a separate paid add-on) to replicate what Postgres does out of the box.
  • Schema flexibility is a weakness here — for our metadata tables (users, files, permissions), we want enforced structure. Mongo's flexibility means bugs silently write wrong shapes.
  • Not what SQLAlchemy is built for — our entire server is written with SQLAlchemy async. Switching means rewriting every model, every query, and every migration.
Summary: The client's MongoDB is their business — it stores their application data and we never touch it. danta-search uses Postgres purely as its own internal brain — storing metadata, embeddings, permissions, and relationships. Postgres gives us vector search, fuzzy search, graph traversal, and strong transactions all in one database. MongoDB would require four separate tools to replicate the same capabilities.

4. Multi-Strategy Retrieval Engine

The retrieval engine is what makes danta-search viable at 1M+ files. Nine sequential stages reduce the candidate pool from 1 million to a fixed budget of 20 files — using every available signal. The LLM never sees more than 20 file summaries.

1
Temporal Parsing  (deterministic — no LLM)
Extract date intent from question. "last week" → {start: Apr 13, end: Apr 19}. "Q1 2025" → {start: Jan 1 2025, end: Mar 31 2025}. "this month" → calendar resolution. Handles relative, absolute, and fiscal references.
↓
2
Domain / Permission Filter  (SQL WHERE clause)
Resolve user's access set from JWT. super_admin: no filter. Others: WHERE container_id IN (:owned_ids + :granted_ids). This runs as a SQL pre-filter — unauthorized files are physically excluded, never even ranked.
↓
3
Date Range Overlap Filter  (SQL, milliseconds)
Only keep files where date_range_start ≤ query_end AND date_range_end ≥ query_start. At 1M files, this typically reduces to 50–500 candidates. This eliminates the "right file is 16th" problem — wrong-period files never reach ranking.
↓
4
BM25 Keyword Search  (Postgres tsvector)
Full-text search on file names, column names, and AI description. Exact term matches score highest. "bonus" finds files with a column literally named "bonus". Best for when the user knows the exact data field name.
↓
5
Fuzzy Match  (pg_trgm trigram similarity)
Catches typos and near-matches. "emplyee" ≈ "employee". "revenu" ≈ "revenue". "dept" ≈ "department". Trigram similarity threshold 0.4. Also handles plural/singular variations. Critical for messy data with inconsistent column naming.
↓
6
Vector / Semantic Search  (pgvector HNSW)
Embed the user question → cosine similarity against file description embeddings. Finds semantically related files even when exact words don't match. "How much did we pay staff?" finds "employee_salary.parquet" without "pay" or "staff" appearing in the file name. Runs on the pre-filtered candidate set — fast.
↓
7
Graph Expansion  (FileRelationship edges)
For each top-ranked file, follow its FileRelationship edges (built at ingest via column-hash matching). Add connected files with confidence > 0.7, up to +5 files. This catches join-partner files (e.g. if you find payroll.csv, graph adds employees.csv that shares emp_id). NOT unlimited expansion — fixed +5 max.
↓
8
Reciprocal Rank Fusion  (combine all scores)
Each signal (BM25, fuzzy, vector, graph) contributes a rank score. RRF formula: score = Σ 1/(k + rank_i) where k=60. Final unified rank list. A file that ranks #3 in BM25 AND #5 in vector beats a file that only ranks #1 in one signal. Prevents over-reliance on single signal.
↓
9
Fixed Budget: Top 20 Files  (hard limit, never grows)
Take top 20 by RRF score. NEVER exceeds 20. If the answer is genuinely not in top 20, the agent surfaces: "Confidence is low. Found these 20 relevant files — did you mean one of these?" — not a silent wrong answer.

Score Contribution by Search Type

Search Type Strength Weakness Weight in Fusion
BM25 Keyword Exact column/file name match Misses synonyms ("pay" ≠ "salary") High (exact match = high confidence)
Fuzzy (pg_trgm) Typos, near-matches, abbreviations Can match unrelated short strings Medium (boost for near-exact)
Vector Semantic Meaning-based, handles synonyms Can miss exact matches if phrased differently High (backbone of conceptual search)
Graph Expansion Finds join-partners you didn't know to ask for Only useful after high-confidence seed files Additive (bonus files, not ranked against core)

5. Permission Model — Client vs Domain Admin vs User

SUPER ADMIN
CLIENT (Organization Owner)
  • Access to ALL domains
  • Access to ALL containers
  • Cross-domain queries
  • Manage domains + containers
  • Grant container access to domains
  • View all user activity + logs
  • Export any file / query result
  • Manage archival settings
DOMAIN ADMIN
Department Manager
  • Access to their domain only
  • Manage users in their domain
  • Grant specific files to other domains
  • Upload files to their containers
  • View their domain's logs
  • Cannot see other domains' data
  • Cannot create new domains
  • Cannot manage global settings
DOMAIN USER
End User / Analyst
  • Access to their assigned domain
  • Access to explicitly granted files
  • Read-only chat queries
  • Export their own query results
  • Use saved query templates
  • Cannot upload files
  • Cannot access other domains
  • Cannot grant access to others

Table: file_domain_access

Revoke = DELETE that row. No data moves. Access is resolved at query time from this table.

How the Filter Flows at Query Time

1. User sends message → JWT decoded → user_id extracted
2. Resolve access → owned_container_ids = [hr_container]
                         granted_container_ids = [finance_shared_file_1] (from file_domain_access)
3. All retrieval queries add: WHERE container_id = ANY(['hr_container','finance_file_1'])
4. Vector search adds the same filter before cosine similarity
5. DuckDB execution only reads Parquet files within that access set
6. Result → user gets data from allowed files only. Physically impossible to see other data.
super_admin bypass: When role = super_admin, Step 3 omits the WHERE clause entirely. All 1M files are candidates. This is the only difference in the code path — a single conditional check.

6. End-to-End Query Flow — Real Example

Finance domain user asks: "What was last week's total bonus payout by department?" — Today is April 22 2026.

STEP 1 — Temporal Parse
"last week" → start: 2026-04-13, end: 2026-04-19

STEP 2 — Domain Filter
Role: domain_user, Finance → access_set = [finance_container, hr_bonus_shared]

STEP 3 — Date Range Filter (SQL)
WHERE date_range_start <= '2026-04-19' AND date_range_end >= '2026-04-13'
1,000,000 files → 47 candidates remain

STEP 4 — BM25 on 47 candidates
"bonus" matches payroll_weekly.parquet (col: bonus), dept_bonuses.parquet → top scores

STEP 5 — Fuzzy on 47 candidates
"department" ≈ "dept" → dept_summary.parquet boosted

STEP 6 — Vector Search on 47 candidates
Embed query → cosine → weekly_compensation.parquet scores high (semantic match)

STEP 7 — Graph Expansion
payroll_weekly.parquet has edge to employees_master.parquet (shared: emp_id, department)
→ employees_master added (+1 file)

STEP 8 — RRF Fusion + Budget
Top 20 by combined score. payroll_weekly.parquet ranked #1, dept_bonuses.parquet #2

STEP 9 — Prompt Assembly
System prompt includes 20 file summaries with schemas + quality notes
Domain context: "Finance user. Date range: Apr 13–19 2026"

STEP 10 — GPT-4 writes SQL
SELECT department, SUM(bonus) as total_bonus
FROM 'az://container/payroll_weekly.parquet'
WHERE date BETWEEN '2026-04-13' AND '2026-04-19'
AND bonus IS NOT NULL
GROUP BY department ORDER BY total_bonus DESC

STEP 11 — DuckDB Execution
{ rows: [{dept: "Sales", bonus: 142000}, {dept: "HR", bonus: 89000}, ...], excluded: 12 }

STEP 12 — Final Response
"Bonus payouts for April 13–19 2026 by department:
  Sales: $142,000 | HR: $89,000 | Finance: $76,000
  Total: $307,000
  Note: 12 rows with missing bonus values were excluded.
  Source: payroll_weekly.parquet (1,842 rows processed)"
Every number came from DuckDB. The LLM only wrote the SQL and formatted the sentences. No hallucination is possible because the values are injected FROM the tool result, not recalled from training weights.

8. CSV Export — Reusable Components

The client asked for the system to expose data as reusable CSV components. This means two things:

A — Instant Query Result Export

Every chat response that contains tabular data has a "Download CSV" button. The DuckDB result is piped through pandas to CSV bytes and sent as a file download. No special setup needed — this works for every query automatically.


How it works: DuckDB returns a dataframe → pandas.to_csv() → FastAPI StreamingResponse with Content-Disposition: attachment.

B — Saved Query Templates (Reusable)

User runs a query → clicks "Save as Template" → gives it a name and parameters. Example: "Monthly bonus report for {department} in {month} {year}".


Next month: open template → fill {department}="Sales", {month}="May", {year}="2026" → click Run → get fresh CSV. Other users in the same domain can subscribe to the same template.


Templates are stored in the saved_queries table. Parameters defined in JSONB. The SQL template uses Python-style {param} substitution with strict allowlist validation to prevent SQL injection.

Reusable component concept: A saved query template is the data-extraction equivalent of a UI component. Define it once. Run it on demand with different parameters. Share it across your team. This is why users don't need to know SQL — they use pre-built templates designed by analysts, parameterized for anyone to run.
