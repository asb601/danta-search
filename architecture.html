1. Executive Summary

danta-search is not a normal chatbot over files. It is a controlled data intelligence backend that converts uploaded ERP and operational files into a searchable, relationship-aware, query-ready semantic layer. The LLM is used for language understanding and SQL assistance, but every number in the final answer must come from validated file schemas and query execution.

Current

Network-first runtime

The VM orchestrates API requests, retrieval, SQL validation, and query execution. Durable data is outside the VM: metadata lives in PostgreSQL, files live in Azure Blob, and LLM calls go to Azure OpenAI.

Current

Grounded answers

The system retrieves relevant files, hydrates real schemas, validates SQL columns, executes against Parquet/CSV files, and returns rows, counts, sources, and CSV downloads.

Future

Business Layer

The planned ingestion layer will tag ERP entities, process steps, status meanings, join rules, date roles, and domain language so query planning becomes business-aware instead of only file-aware.

The platform reduces VM burden because the VM is an orchestrator — it makes network calls to managed services and file storage instead of storing everything locally. For production performance, compute, database, and object storage should be in the same cloud region.

2. Current Architecture

The current system is separated into user experience, backend orchestration, metadata control plane, object storage, LLM services, and runtime SQL execution. That separation is the main strength: the VM does not need to hold all customer data on disk, and the backend can scale by improving retrieval, metadata, and execution without changing the client UI.

What the VM does

  • Receives API requests from the UI.
  • Authenticates users and resolves container access.
  • Coordinates ingestion jobs.
  • Builds retrieval shortlists and agent prompts.
  • Runs SQL execution against remote Parquet/CSV files.
  • Streams final answers and tables to the user.

What is outside the VM

  • Uploaded business files are stored in Azure Blob.
  • Metadata, embeddings, relationships, and audit data are stored in PostgreSQL.
  • LLM reasoning and embeddings are served by Azure OpenAI.
  • CSV/Parquet source data remains in object storage, not local VM disk.

Network calls reduce the storage burden on the VM, but SQL execution still uses VM CPU, memory, and network bandwidth when DataFusion or DuckDB scans remote files. The VM is lighter than a database server and does not store client data locally, but it does perform non-trivial query processing work.

3. Current Ingestion Pipeline

Ingestion is the foundation of answer quality. The system does not wait until question time to understand files. It profiles, cleans, describes, embeds, and relates files as they enter the platform.

1

Upload or connect file container

The user uploads files or configures an Azure Blob container. The backend stores the encrypted connection string in container_configs and scopes every file to that container.

2

File validation and parsing

The ingestion job reads CSV, Excel, Parquet, and delimiter-based text files. It handles encoding, delimiters, sheets, and file-level parse errors before exposing the file to chat.

3

Cleaning and quarantine audit

Rows that look empty, malformed, or non-data are tracked in file_analytics.cleaning_audit, quarantine_count, and quarantine_sample. This gives the demo a quality story, not just an AI story.

4

Parquet conversion

Query-ready Parquet paths are stored in file_analytics.parquet_blob_path. Parquet is preferred at query time because it is columnar and faster to scan than raw CSV.

5

Profiling and analytics

The backend calculates row counts, column counts, column statistics, value counts, and cross-tabs. These go into file_metadata and file_analytics so many answers can be supported from precomputed evidence.

6

AI description and retrieval text

The system generates file descriptions, good_for hints, key metrics, key dimensions, search text, and embeddings. This lets users ask business questions without knowing exact file names.

7

Semantic role detection

Columns are tagged with roles such as entity keys, measures, dates, statuses, dimensions, and ERP-specific identifiers. These roles are stored in file_metadata.column_semantic_roles with evidence and confidence signals.

8

Join-key registry and relationship detection

The platform stores fingerprinted key evidence in column_key_registry and creates file_relationships when columns have enough semantic and value-overlap evidence to support a join.

9

Semantic layer build

Files become semantic entities and approved relationships in semantic_entities and semantic_relationships. This is the start of the business graph used during question answering.

The backend does not simply store uploaded files. It creates a governed metadata layer of the data estate: what each file means, what columns exist, which columns join, which values are dirty, and which files are safe to use together. This metadata is the foundation of reliable query results.

4. Current Querying Pipeline

The query path is built to avoid the classic failure mode of LLM analytics: choosing the wrong table, inventing columns, and then giving confident but false numbers. The current design separates natural-language understanding from retrieval, schema validation, SQL execution, and answer formatting.

1

Question received

The chat API receives the natural-language question and resolves user, role, container, and conversation context.

2

Intent and entity planning

The planner extracts durable business objects, time constraints, filters, and requested behaviors. Example: a PO summary question becomes entity purchase_order, year 2025, and summary facets.

3

Entity resolution

The resolver maps business entities to candidate files using table names, acronyms, semantic roles, and confidence scoring. This finds files such as PO headers, PO lines, invoice lines, distributions, and receipts without hardcoding one question.

4

Multi-strategy retrieval

The retrieval engine combines BM25, fuzzy matching, vector search, graph evidence, semantic recovery, and reciprocal-rank fusion. The output is a focused shortlist, not the entire catalog.

5

Schema hydration and prompt context

The selected files are expanded with real column names, semantic roles, date columns, join hints, and approved relationships. The LLM sees actual schemas instead of guessing.

6

Deterministic summary fast path

For entity-plus-year summary questions, the backend now builds the aggregate directly from inspected schemas. This prevents the agent from drifting into invoice-only or delivery-only answers when the user asked for a PO lifecycle summary.

7

LangGraph agent for open-ended analysis

For harder questions, the LangGraph agent can call tools such as search_catalog, get_file_schema, inspect_column, extract_relations, run_sql, and summarise_dataframe.

8

SQL validation and canonicalization

Logical table names are canonicalized to authorized Azure Blob paths. SQL is checked against inspected schemas so invented columns and unsafe aliases are rejected before execution.

9

Execution and evidence-based answer

DataFusion or DuckDB executes the query against Parquet/CSV files in object storage. The final answer returns a narrative, table rows, row counts, source files, and CSV export.

Query Execution Engine: Apache DataFusion and DuckDB

The platform uses two embedded analytical query engines to execute SQL directly against files stored in Azure Blob Storage. Neither engine requires a separate database server — they run inside the backend process and query Parquet or CSV files in place, without copying data to VM disk first.

Primary

Apache DataFusion

What it is: an in-process analytical query engine built in Rust, backed by the Apache Arrow columnar memory format. It is the same execution layer used inside Apache Spark and Delta Lake at the columnar level, but designed to run as an embedded library rather than a cluster.

Why it is the primary engine: DataFusion creates a fresh, isolated SessionContext for every query. This means 40 concurrent user queries run truly in parallel — no shared mutable state, no queuing, bounded only by CPU cores and Azure Blob read bandwidth. DuckDB's thread-local connection model serializes under concurrent load; DataFusion does not.

How it accesses files: it registers Azure Blob as a native object store and reads Parquet files directly over HTTPS without downloading them. SQL references like read_parquet('az://container/file.parquet') are rewritten internally to registered table handles before execution.

  • Columnar vectorized execution (Arrow format)
  • Per-query isolated session — no cross-request state leakage
  • Predicate pushdown — filters applied at read time, not after loading all rows
  • Native Azure Blob integration via object store API
Fallback

DuckDB

What it is: an embedded OLAP (Online Analytical Processing) database engine optimized for fast analytical queries on files. It is often described as "SQLite for analytics" — a single-file, zero-server engine that can query CSV, Parquet, JSON, and Arrow formats directly.

Why it is kept as a fallback: DuckDB has broader SQL dialect compatibility and handles some edge-case file formats more gracefully. It is used when DataFusion returns an error, or when the configured query engine is set to duckdb explicitly.

Limitation under concurrency: DuckDB uses thread-local connections with shared internal structures. Under high concurrent query load, queries begin to serialize — each query waits for the previous one's internal locks to release. For single-user or low-concurrency scenarios this is not an issue, but it is why DataFusion is preferred at scale.

  • Excellent single-query performance on structured files
  • Broad SQL dialect and function support
  • Zero server process — runs entirely in-process
  • Used as graceful fallback if DataFusion execution fails

What both engines share: neither engine copies files to local disk before querying. Both scan Parquet columns from Azure Blob over the network, applying column pruning and predicate pushdown to minimize data transfer. The validated SQL is executed against the actual uploaded business data — the LLM never sees or touches the raw rows. Query results (aggregates, counts, sampled rows) are what the LLM uses to compose the final answer.

Bounded execution: every query runs inside a concurrency-controlled executor with a hard per-query timeout. If a query exceeds its time budget it is cancelled and the user receives an explicit timeout error rather than a stalled connection. This prevents a single large file scan from blocking all other active users.

5. Network and Cloud Placement

The current design works through network calls: the browser calls the VM, the VM calls PostgreSQL, Azure Blob, and Azure OpenAI, and the SQL engine scans remote files. That is good architecture because it avoids putting every asset on VM disk. But cloud placement still matters a lot.

LLM and metadata calls are offloaded to managed services, but SQL scans still consume VM network and compute. For large Parquet file scans, cloud co-location and query-engine sizing have a direct impact on response time and cost.

6. Future Architecture: Business Layer in Ingestion

The next major architectural improvement is a Business Layer created during ingestion. Today the system understands files, columns, roles, and relationships. The future layer will understand ERP meaning: business domains, table purpose, process stages, status polarity, document lifecycle, and client-specific terminology.

Current ingestion
File -> clean/profile -> metadata -> analytics -> relationships -> semantic layer

Future ingestion
File -> clean/profile -> metadata -> analytics -> relationships
     -> Business Layer builder
     -> ERP tags, process tags, status semantics, join contracts, glossary
     -> Query-time business context packet
ERP tags

Recognize business objects

Identify whether a file is purchase order, invoice, vendor, receipt, GL journal, customer, delivery, inventory, payroll, asset, project, or another ERP object.

Process tags

Understand workflows

Map files into workflows such as procure-to-pay, order-to-cash, record-to-report, inventory, fixed assets, projects, and HR/payroll.

Status semantics

Store meaning, not labels

Record whether approved_flag = Y means approved, not pending. This prevents false conclusions from positive/negative status fields.

Join contracts

Govern table relationships

Promote relationships from candidate joins into approved business join contracts with key columns, cardinality, confidence, and risk notes.

Facet evidence

Know what can be answered

For a summary request, the layer records which files and columns support status, approval, delivery, invoice matching, payment, aging, quantity, and amount facets.

Client glossary

Adapt to local language

Store client-specific names such as PO, PR, GRN, vendor, supplier, invoice hold, clearing document, shipment, or delivery note.

7. Future Querying Architecture

With the Business Layer, query planning becomes more deterministic. The system will not only ask, "which files look relevant?" It will ask, "which business object, workflow, date filter, and evidence facets are required?"

1

Human question

User asks in normal language: "Analyze PO details for 2025: status, pending approvals, delivery, invoice matching, next actions."

2

Business intent packet

The system extracts subject purchase_order, year 2025, workflow procure_to_pay, and facets: current status, approvals, delivery, invoice matching, recommendation.

3

Facet coverage check

The Business Layer checks whether each facet has evidence columns. If invoice matching columns are missing, the answer says not verifiable instead of inventing a clean result.

4

Approved SQL scaffold

The planner uses approved join contracts and status polarity rules to generate SQL. For common summaries, deterministic templates can be used before the LLM is involved.

5

Evidence answer

The final answer includes numbers, source files, missing-evidence notes, and recommended actions. The client can trace every claim back to a table, column, and query result.

The platform will advance from file search to business workflow intelligence. A PO lifecycle question will automatically map to headers, lines, shipments/receipts, invoices, holds, and payments — and will verify which of those facets are supported by the uploaded data before answering.

9. How Answers Are Produced

Every answer produced by danta-search is traceable to specific database records, validated schemas, and executed queries. The system does not generate answers from model memory. Numbers, counts, and status summaries come from query execution over real uploaded files.

Grounded file selection

The system selects files using multi-strategy retrieval against file_metadata, column_semantic_roles, and the semantic relationship graph. The LLM receives focused, validated schemas — not an open-ended prompt over all files.

Evidence-backed relationships

Joins between files are derived from column_key_registry, file_relationships, and semantic_relationships. Each join has a confidence score, value overlap percentage, and semantic role match that can be inspected.

Schema-validated SQL

Generated SQL is checked against inspected schemas and authorized Azure Blob paths. Column names and table references are validated before execution — invented columns are rejected, not silently executed.

Query-executed results

All numerical results, row counts, and aggregates are produced by DataFusion or DuckDB executing against Parquet and CSV files in Azure Blob Storage. The final answer includes source file citations and an exportable result table.

Answer traceability

Each answer in the conversation log references the files used, the schemas inspected, the SQL executed, and the execution row count. Where evidence is incomplete — for example, an invoice matching status column is absent from the uploaded data — the system reports that gap explicitly rather than producing a speculative answer.
